API Overview

This document describes jKool Track & Trace API (TNT4J) specification for tracking and tracing application activities and behavior for rapid diagnostics and root cause analysis. The framework encapsulates the following key elements essential during diagnostics:

·           Performance, response time

·           Exceptions, faults

·           Application state, dumps

·           Message flow (send->receive)

·           Granular conditional logging based on application specific tokens/keys

·           Correlation within and across multiple application components, runtime environments

Track & Trace 4J (TNT4J) API

TNT4J allows developers to imbed custom tracking and tracing code very quickly into java applications. It is recommended that developers use this API when instrumenting Java applications with track and trace instrumentation. TNT4J delivers the following features aimed at delivering fast diagnostics and root-cause analysis:

 

·           Simple programming model to facilitate fast root-cause, log analysis

·           Automated timing of application activities and sub-activities (elapsed, idle time, message age)

·           Application state dump framework for reporting internal variables, data structures

·           Conditional logging based on application tokens, patterns

·           Inter-log correlation of log entries (correlators and tags) between multiple related applications

·           Intra-log correlation of related activities and sub-activities between multiple applications and threads

·           Event location tags such as GPS, server etc.

·           Message flow direction for composite applications that exchange messages (e.g. SOAP, JMS, and SQL etc.)

·           User defined properties such as CPU, memory logging, thread statistics per process/thread

·           Granular context such as thread id, process id, server, application name

 

TNT4J introduces the following constructs:

 

·         Application Activity (TrackingActivity) – a logical grouping of related events and messages, which has start/end time

·         Tracking Event (TrackingEvent) – an event emitted about a step in processing or execution, which has severity, start/stop time, location, correlator as well as message payload (text)

·         Event Sink (EventSink) – a generic destination where activities and events are routed for recording such as event log, or central service that analyzes log entries.

·         Dump Provider (DumpProvider) – application supplied dump handlers that generate collections of key/value pairs describing application’s state, internal variables, data structures.

·         Snapshot (Snapshot) – a collection of items such as properties, metrics. Snapshots are used to report user defined properties associated with events.

 

Use TrackingLogger to interface with all Tracker logging features. TrackingLogger is a helper class with static calls to Tracker logging interface. Application should use this helper class instead of obtaining a Tracker logger instance per thread using TrackerFactory. TrackingLogger obtains the Tracker logger instance and stores it in thread local associated for each thread. A TrackingEvent represents a specific tracking event that application creates for every discrete event such as JDBC, JMS, SOAP or any other relevant application logic. Application developers must obtain a Tracker instance via TrackerFactory, create instances of TrackingEvent and use TrackingLogger.tnt() to report tracking activities. Alternatively use TrackerFactory directly to interface with logging facilities without the need to obtain a logger instance. Simply register application thread using TrackingLogger.register() call and report tracking events as shown in the example below.

 

NOTE: Inter-thread or inter-application/process activities are tied via TrackingEvent correlators. Make sure to use TrackingEvent correlators as a way to interconnect related inter-application activities.

 

A TrackingActivity start()/stop() methods calls used to mark application activity boundaries. Application may create instances of TrackingEvent to time individual sub-activities and report them using TrackingActivity.tnt()method call.

 

Typical application logic:

TrackerConfig config = DefaultConfigFactory.getInstance().getConfig("com.nastel.appl.name").build();

TrackingLogger tracker = TrackingLogger.getInstance(config); // register and obtain Tracker logger instance

TrackingActivity activity = tracker.newActivity(); // create a new application activity

activity.start();

TrackingEvent event = tracker.newEvent(OpLevel.SUCCESS, "SQL-SELECT", "SQL customer lookup"); // create a tracking event

event.start(); // start timing a tracking event

try {

        ...

        ...

        event.stop(); // stop timing tracking event

} catch (SQLException e) {

        event.stop(e); // stop timing tracking event and associate an exception

        ...

} finally {

        activity.stop(); // stop application activity, timing

        activity.tnt(event); // track and trace tracking event, associate tracking event with the given activity

        tracker.tnt(activity); // track and trace application activity

}

TrackingEvent implements tracking for a specific application sub-activity. Applications create instances of TrackingEvent after creating and starting an application activity using TrackingLogger.newActivity() and TrackingActivity.start(). Each TrackingEvent can be timed independently using start()/stop() method calls and reported and logged using TrackingActivity.tnt() method call.

 

Event Correlators and Locators

Any pair of TrackingEvent instances is related when event correlators match. TrackingEvent A is related to TrackingEvent B when A.getCorrelator() == B.getCorrelator(). Developers may use signatures and or correlators to correlate tracking events across application/server boundaries. TrackingEvent A is the same as TrackingEvent B if the corresponding event signature are identical: meaning A.getSignature() == B.getSignature(). Currently each TrackingEvent gets assigned a unique event signature using TrackingEvent.newUUID() call. Signatures can be changed using TrackingEvent.setSiganture() method call. Signatures and correlators must be set before making Tracker.tnt()call. Event location can be any string that identifies location of the tracking event such as GPS, server or any other useful locator.

 

Below is an example of how to set an event correlator and event location to GPS coordinates (getGPS() is a hypothetical call to obtain GPS coordinates):

 

TrackerConfig config = DefaultConfigFactory.getInstance().getConfig("com.nastel.appl.name").build();

TrackingLogger tracker = TrackingLogger.getInstance(config); // register and obtain Tracker logger instance

TrackingActivity activity = tracker.newActivity(); // create a new application activity

activity.start();

TrackingEvent event = tracker.newEvent(OpLevel.SUCCESS, "SQL-SELECT", "SQL customer lookup"); // create a tracking event

event.setCorrelator(“7271292”); //set event correlator to some token that should be used across thread and application boundaries

event.start(); // start timing a tracking event

try {

        ...

        ...

        event.setLocation(getGPS()); // obtain GPS location and set event location

        event.stop(); // stop timing tracking event

} catch (SQLException e) {

        event.stop(e); // stop timing tracking event and associate an exception

        ...

} finally {

        activity.stop(); // stop application activity, timing

        activity.tnt(event); // track and trace tracking event, associate tracking event with the given activity

        tracker.tnt(activity); // track and trace application activity

}

Correlating Sender/Receiver Activities 

Applications may also indicate whether the tracking event signifies SEND/RECV/INQUIRE or other types of exchange by using OpType when creating TrackingEvent instances.

Below is an example of specifying operation types associated with the tracking event:

 

Sender application code (SENDER):

TrackerConfig config = DefaultConfigFactory.getInstance().getConfig("com.nastel.appl.Sender").build();

TrackingLogger tracker = TrackingLogger.getInstance(config); // register and obtain Tracker logger instance

TrackingActivity activity = tracker.newActivity(); // create a new application activity

activity.start(); 

 

TrackingEvent event = tracker.newEvent(OpLevel.INFO, OpType.SEND, "SendOrder”, “7271292”, "Sending Purchase Order"); // create a tracking event with a correlator and an associated operation type

event.start(); // start timing a tracking event

try {

        ...

        ...

        event.setLocation(getGPS()); // obtain GPS location and set event location

        event.stop(); // stop timing tracking event

} catch (SQLException e) {

        event.stop(e); // stop timing tracking event and associate an exception

        ...

} finally {

        activity.stop(); // stop application activity, timing

        activity.tnt(event); // track and trace tracking event, associate tracking event with the given activity

        tracker.tnt(activity); // track and trace application activity

}

 

Receiver application code (RECEIVER):

String order_id = “7271292”;

Throwable error = null;

TrackerConfig config = DefaultConfigFactory.getInstance().getConfig("com.nastel.appl.Receiver").build();

TrackingLogger tracker = TrackingLogger.getInstance(config); // register and obtain Tracker logger instance

TrackingActivity activity = tracker.newActivity(); // create a new application activity

activity.start(); 

 

TrackingEvent event = tracker.newEvent(OpLevel.INFO, OpType.RECEIVE, "RecvOrder", order_id, "Receiving order"); // create a tracking event with a correlator and an associated operation type

event.start(); // start timing a tracking event

try {

        ...

        ...

        order_id = response.getOrderId(); // sample code to obtain order id from the received response message

        event.setLocation(getGPS()); // obtain GPS location and set event location

        event.setCorrelator(order_id); // assign a correlator from order_id

        event.stop(); // stop timing tracking event

} catch (SQLException e) {

        error = e;

        event.stop(e); // stop timing tracking event and associate an exception

        ...

} finally {

        activity.stop(ActivityStatus.EXCEPTION, e); // stop application activity, timing, associate an exception

        activity.tnt(event); // track and trace tracking event, associate tracking event with the given activity

        tracker.tnt(activity); // track and trace application activity

}

 

Both sender and receiver are not only correlated via a correlator “7271292” but the direction of communication is established as well from SENDER->RECEIVER. This level of logging allows log analysis tools to show related activities as well as graph of activity flow from sender to receiver. It is very useful for applications that actually exchange information or messages in one or both directions.

 

Event Tags vs. Correlators

Applications may also set message tags that could be used to find related activities. Message tags are user defined string labels that can be used in addition to correlators. The difference between tags and correlator is that that tags are not used to tie TrackingEvent instances together. Tags are used just as string labels associated with a particular TrackingEvent instance. Below is an example of the code to set a message tag:

 

String order_id = “7271292”;

Throwable error = null;

TrackerConfig config = DefaultConfigFactory.getInstance().getConfig("com.nastel.appl.Receiver").build();

TrackingLogger tracker = TrackingLogger.getInstance(config); // register and obtain Tracker logger instance

TrackingActivity activity = tracker.newActivity(); // create a new application activity

activity.start(); 

 

TrackingEvent event = tracker.newEvent(OpLevel.INFO, OpType.RECEIVE, "RecvOrder", order_id, "Receiving order"); // create a tracking event with a correlator and an associated operation type

event.start(); // start timing a tracking event

try {

        ...

        ...

        order_id = response.getOrderId(); // sample code to obtain order id from the received response message

        event.setCorrelator(order_id); // assign a correlator from order_id

        event.setLocation(getGPS()); // obtain GPS location and set event location

        event.setTag(response.getOrderInfo()); // assign message tag

        event.stop(); // stop timing tracking event

} catch (SQLException e) {

        error = e;

        event.stop(e); // stop timing tracking event and associate an exception

        ...

} finally {

        activity.stop(ActivityStatus.EXCEPTION, e); // stop application activity, timing, associate an exception

        activity.tnt(event); // track and trace tracking event, associate tracking event with the given activity

        tracker.tnt(activity); // track and trace application activity

}

 

Activity Listeners

Developers can use activity listeners ActivityListener to be notified when activities are started and stopped. Below is an example of an ActivityListener implementation. These listeners can be used for pre and post processing of tracking activities before they are actually reported.

 

class MyActivityHandler implements ActivityListener {

      @Override

    public void started(Activity activity) {

            System.out.println("activity.id=" + activity.getTrackingId()

                        + ", activity.name=" + activity.getName()

                        + ", started=" + activity.getStartTime());

      }

 

      @Override

    public void stopped(Activity activity) {

            // post processing of activity: enrich activity with application metrics

            PropertySnapshot snapshot = new PropertySnapshot("APPL_METRICS");

            snapshot.add("appl.activity.count", activityCount);

            snapshot.add("appl.event.count", eventCount);

            activity.add(snapshot); // add property snapshot to activity

            System.out.println("activity.id=" + activity.getTrackingId()

                        + ", activity.name=" + activity.getName()

                        + ", elapsed.usec=" + activity.getElapsedTime()

                        + ", snap.count=" + activity.getSnapshotCount()

                        + ", item.count=" + activity.getItemCount());

    }

}

 

Activity listeners can be registered using the following example:

            // register with the TNT4J framework

            TrackerConfig config = DefaultConfigFactory.getInstance().getConfig(args[0]);

            config.setActivityListener(new MyActivityHandler());

            TrackingLogger tracker = TrackingLogger.getInstance(config.build());

 

Activity listener will be called for every activity instance on start and stop.

Sink Listeners

Sink listeners can be used to intercept sink errors (SinkErrorListener), events (SinkLogEventListener) and filter out events (SinkEventFilter) written to the underlying event sink. Each of the listener instances can be registered using the following methods.

            // register with the TNT4J framework

            TrackerConfig config = DefaultConfigFactory.getInstance().getConfig(args[0]);

            config.setActivityListener(new MyActivityHandler());

            TrackingLogger tracker = TrackingLogger.getInstance(config);  

            TrackingLogger.addSinkErrorListener(new MySinkErrorHandler());

            TrackingLogger.addSinkLogEventListener(new MySinkLogHandler());

            TrackingLogger.addSinkEventFilter(new MySinkEventFilter());

 

            // SinkErrorListener is called when a failure writing to the sink occurs

class MySinkErrorHandler implements SinkErrorListener {

      public void sinkError(SinkError event) {

          System.out.println("onSinkError: " + event);

          if (event.getCause() != null) event.getCause().printStackTrace();        

      }

}

 

            // SinkLogEventListener is called on every event that is actually written to the sink

            // won’t be called if filtered out using SinkEventFilter

class MySinkLogHandler implements SinkLogEventListener {

      public void sinkLogEvent(SinkLogEvent event) {

          System.out.println("sink.LOG: sev=" + event.getSeverity() + ", source=" + event.getSource() + ", msg=" + event.getLogObject());

      }

}

 

            // SinkEventFilter is called on every event. The filter determines if the event is accepted or not by the sink

class MySinkEventFilter implements SinkEventFilter {

      public boolean filter(EventSink sink, TrackingEvent event) {

         boolean pass = event.getSeverity().ordinal() > OpLevel.INFO.ordinal()? true: false;

         System.out.println("sink.FILTER: accept=" + pass + ", sev=" + event.getSeverity() + ", sink=" + sink);

         return pass;

      }

      public boolean filter(EventSink sink, TrackingActivity event) {

      }

      public boolean filter(EventSink sink, OpLevel level, String msg, Object…args) {

      }

}

 

LOG4J Log File Structure and Analysis 

Below is an example of what actually ends up in log4j log file. All related activities are correlated via the activity signature (Signature:87a5c1ce-f46c-4932-aa68-ce026cae07c7). Activity starts with a message having (Status:BEGIN) and ends with (Status:END) with the same process and thread ID (PID:17584,TID:22) . Each activity end message will also include a set of properties measured between activity start/stop() calls such as (Property(Type:6,Name:CPU_USAGE_USEC,Value:3400821.8,SampleTime:[10:25:34.767000])), which shows  the number of CPU microseconds used during execution of the given activity as well as the time the measurement was sampled (SampleTime:[10:25:34.767000]). These properties are measured per thread per activity and can be used to identify activities with high CPU/memory utilization. TNT4J will also measure cumulative thread block and wait statistics at the end of each activity using (Property(Type:XX,Name:BLOCKED_XXX, Property(Type:XX,Name:WAITED_XXX) properties. Both sender and receiver logs can be correlated via Correlator:7271292 field search. Related tracking events within each log can be correlated via grouped Signature:XXXXX searches (e.g. all tracking events with Signature:87a5c1ce-f46c-4932-aa68-ce026cae07c7). All activities by process/thread-id can be related by (PID:XXXXX,TID:XX) search. Each application activity TrackActivity and sub-activity TrackingEvent elapsed times are identified by (ElapsedUsec:XXX)field. Applications that exchange messages as in this example, SENDER->RECEIVER activities which designate SEND and RECEIVE can be marked using OpType when creating TrackingEvent instances. Such markers are shown in the log as (Type:SEND or Type:RECEIVE). Event location markers are identified by (Location:XXX) field and can be any string locator associated by the application.
